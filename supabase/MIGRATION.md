## 실행 순서

**001 → 002 → 003 → 004 → 005 → 006** 순서대로 SQL Editor에서 실행.
각 파일 끝에 검증 쿼리가 있으니 결과를 확인하고 다음으로 넘어갈 것.
001~006은 **기존 배포 앱을 깨뜨리지 않는다** (컬럼 추가만, 삭제·개명 없음).

| 파일 | 하는 일 | 검증 포인트 |
|---|---|---|
| 001 | profiles를 auth와 분리 + 김기사·이기사·차호근 프로필 생성 | 프로필 5행 |
| 002 | units 신설 + sites 데이터로 백필 | 현장별 호기 수 = unit_count |
| 003 | site_assignments 신설 + 주담당 백필 | 미배정 현장 0 |
| 004 | self_checks 신설 + 월별 생성 함수 | (테이블만, 데이터는 함수 호출 시) |
| 005 | 기록 테이블에 unit_id, *_id 병행 컬럼 추가 | 컬럼 목록 출력 |
| 006 | 라벨→unit_id, 이름→profile id 변환 (재실행 안전) | **미매칭 전부 0 확인** |

**007은 지금 실행 금지** — 앱 코드 전환·배포·실사용 검증(1~2주) 후에.

## 006 미매칭이 나오면

- `todos unit 미매칭`: 원천(자재/견적)도 라벨도 없는 manual 할일 → 정상(현장 무관 할일). 라벨이 있는데 미매칭이면 상세 쿼리(파일 하단 주석)로 확인 후 수동 UPDATE.
- `동일빌딩` 관련: 같은 이름 현장이 2곳이라 이름 기반 자동매칭에서 의도적으로 제외됨. billings/restock에 동일빌딩 건이 생기면 수동 지정 (2026-07-15 기준 해당 없음 — 전부 M595타워).
- 이름 미매칭: 새 이름이 등장한 것 → profiles에 행 추가 후 006 재실행.

## 이미 끝나 있어서 안 하는 것 (설계서와 다른 점)

- **사진 마이그레이션 불필요**: photo_urls(배열)·supply_photo_url(s)·before/after/confirm_photo_url이 이미 실DB에 존재하고 실업로드 중 (전임자가 2026-07-15 구현). v2의 `photos text[]` 신설 대신 기존 컬럼명 유지.
- **컬럼 개명 안 함**: arrival_time→arrived_at, error_code→symptom 등은 하지 않음. 화면 코드의 매퍼(lib/mappers.js)가 이름 차이를 흡수 — DB 개명은 위험 대비 이득 없음.

## 결정 기록 (v2.1 보완, DESIGN-v2.md 11장과 동일)

1. profiles.id는 자체 uuid, 로그인 연결은 `auth_user_id` 별도 컬럼 (계정 없는 기사도 프로필 보유 가능). **Phase 2 로그인 켤 때 앱은 `auth_user_id`로 프로필 조회해야 함.**
2. '관리자' 이름 기록은 '관리자(신석주)' 프로필로 병합 (개발용 가짜 이름이었음).
3. uuid PK는 신설 테이블만. 기존 테이블 text PK 유지.
4. todos에 unit_id(null 허용) 추가 — manual 할일도 현장 연결 가능.
5. units에 seq(순번) 컬럼 — 라벨 변환·정렬 기준, 영구 유지.
6. 호기 라벨 두 형식('1-N', 'N호기') 모두 seq로 변환.
