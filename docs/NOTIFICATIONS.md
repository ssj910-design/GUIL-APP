# 알림 설계 (2026-07-20 초안)

## 왜 상황부터 모으는가

알림은 한 번 시끄러워지면 사람들이 전부 꺼버린다. 그러면 정작 급한 것도 안 간다.
그래서 **무엇을 · 누구에게 · 얼마나 급하게 · 어떤 통로로** 를 먼저 정하고 만든다.

## 지금 있는 것 / 없는 것

| | 상태 |
|---|---|
| 앱 안 종 아이콘 알림 | ✅ 있음 (고장·할일·자재·견적·상비·우리방 6종, ElevatorFieldApp) |
| 앱 안 팝업 (근무 교환) | ✅ 있음 — 단, **앱이 열려 있을 때만** |
| **웹 푸시 (앱 꺼져 있어도 옴)** | ❌ **없음** — 서비스워커·VAPID·구독 저장 전부 미구현 |
| 문자(SMS) | ❌ 없음 (시뮬레이션만: lib/sms.js) |

→ 지금 "알림"은 전부 **앱을 켜야 보이는 것**이다. 진짜 알림은 웹 푸시부터 만들어야 한다.

## 발송 주체가 두 가지로 갈린다 (중요)

우리 앱은 백엔드 없이 클라이언트가 Supabase에 직접 쓴다. 그래서 알림 트리거를 이렇게 나눈다.

- **즉시형** — 사람이 행동한 직후. 그 행동을 처리하는 곳에서 `/api/push` 를 함께 호출한다.
  (예: 배정 버튼을 누른 그 순간)
- **시간형** — 날짜가 되어서 알려야 하는 것. **Vercel Cron**이 매일 돌면서 대상을 찾아 보낸다.
  (예: 점검일 3일 전) — 이미 도는 크론이 2개 있으니 같은 방식.

## 급함 등급이 뜻하는 것

- 🔴 **즉시** — 소리·진동 켜서 푸시. 앱이 꺼져 있어도 반드시 간다.
- 🟡 **보통** — 푸시는 가되 조용히(무음). 앱 종 배지에도 남는다.
- 🟢 **낮음** — 푸시 없이 앱 종 배지만. 사용자가 켜면 푸시로 승격.

기본값을 이렇게 두면 하루에 오는 소리 알림이 몇 건으로 줄어 "다 꺼버리는" 사태를 막는다.

## 알림 설정 (마이페이지)

브라우저 알림 권한은 **사이트 단위 하나뿐**이라 종류별 on/off를 브라우저가 못 해준다.
우리가 직접 만들어야 한다.

- 저장: `profiles.notify_prefs jsonb` — `{"failure_assigned": true, "duty_swap": true, ...}`
- 발송 직전 서버에서 이 값을 확인하고 거른다
- 기본값은 위 급함 등급대로 (🔴🟡 켜짐, 🟢 꺼짐)

## 만드는 순서

1. **웹 푸시 기반** — 서비스워커, VAPID 키, `push_subscriptions` 테이블, `/api/push` 발송 라우트
2. **🔴 즉시형 4개** — A-1 배정됨 / A-2 미배정 발생 / B-1 접수됨 / B-3 출동거부
   → 이것만 돼도 고장 대응 속도가 달라진다
3. **알림 설정 화면** (마이페이지) + 종류별 on/off
4. **🟡 즉시형** — 교환·연차·자재 (A-4~6, B-5~6)
5. **시간형 크론** — 점검 임박, 당직 전날, 미배정 지연, 계약 만료
6. SMS 승격 — 🔴인데 5분간 확인 안 하면 문자 (나중)

## 참고 — PWA 설치

푸시를 받으려면 안드로이드 크롬에서 **홈 화면에 추가**(WebAPK 설치)가 사실상 필수다.
manifest는 이미 있으므로([app/manifest.js](../app/manifest.js)) 서비스워커만 추가하면 설치형 앱이 된다.
스토어 등록 없이 배포 가능하고, 앱 업데이트는 우리가 배포하면 자동 반영된다.
